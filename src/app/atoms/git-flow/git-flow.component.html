Git Flow es un modelo de flujo de trabajo basado en Git que define una estructura clara para gestionar ramas en un proyecto. Fue propuesto por Vincent Driessen y es muy útil en equipos grandes o proyectos con lanzamientos frecuentes.

Se basa en el uso de ramas específicas con propósitos definidos, lo que facilita la colaboración y la organización del código.

En Git Flow tenemos 6 tipos de ramas principales:

    Ramas públicadas en ambientes:

  • master o main

    Es la rama principal y contiene código listo para producción. Nunca se trabaja directamente en esta rama.

  • development

    Es la rama base donde se integran nuevas características antes de ir a producción. Contiene código funcional pero aún no completamente probado. Todas las nuevas funcionalidades parten de aquí.

  • release

    Es la rama donde se realizan las pruebas de las funcionalidades, esta rama es la que se fusiona en main para subir a producción. Contiene código funcional y probado.

    • Ramas creadas por el equipo de desarrollo:

  • feature

    Esta rama parte de la rama development y es la que se usa para desarrollar nuevas funcionalidades, generalmente la acompaña un ID de ticket o de HU, por ejemplo feature/HU-123

  • bugfix

    Esta rama parte de la rama release y es la que se usa para corregir errores, generalmente la acompañan un ID de ticket o de HU, por ejemplo bugfix/BUG-123 Cuando se realizan las correciones debe integrarse con release y development

  • hotfix

    Esta rama parte de la rama main y es la que se usa para corregir errores de producción, generalmente la acompañan un ID de ticket o de HU, por ejemplo hotfix/BUG-123 Esta rama se integra con main, release y finalmente con development

Existen muchos más módelos de flujo de trabajo, como Trunk-Based, Feature-Based, OneFlow, etc., Depende de la necesidad de tu equipo.
