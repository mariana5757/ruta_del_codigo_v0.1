En git podemos gestionar, rastrear y revertir cambios, y muchas cosas más. Para ello vamos a revisar algunos comandos basicos de git, que te ayudarán a gestionar tu proyecto de manera eficiente.

  • git clone URL-del-repositorio

    Crea una copia local de un repositorio en tu computadora. URL-del-repositorio es la URL del repositorio que quieres clonar. Cuando ya tengas la copia local, si tienes permisos, puedes empezar a trabajar con ella y colaborar con otros desarrolladores o subir cambios al repositorio original.

  • git init

    Inicializa un repositorio Git en tu proyecto. Creando una carpeta oculta llamada .git, donde se almacenan los archivos de control de versiones.

  • git status

    Verifica el estado de tu proyecto. Muestra los archivos modificados, no modificados, o los archivos que no existen en el repositorio.

    Por ejemplo en la imagen vemos los archivos donde dice "changes to be committed", son aquellos que han sido modificados y añadidos al área de preparación (staging area) con el comando git add. Estos cambios están listos para ser incluidos en el siguiente commit, pero aún no han sido confirmados (committed).

    Tenemos los archivos "changes not staged for commit" son aquellos que han sido modificados pero no se han añadido al área de preparación, por lo que no serán parte del próximo commit a menos que uses git add para incluirlos.

    Y tenemos "untraked files" son aquellos que no han sido modificados y no se han incluido en el área de preparación, Git te está diciendo que hay archivos en tu directorio que aún no han sido añadidos al área de preparación (staging area).

  • git log --oneline --graph --all

    Muestra un resumen de los commits realizados en el repositorio de forma compacta y en árbol.

    En la imagen vemos un ejemplo de un resumen de commits, cada linea representa un commit.

    • * f9caa50 (HEAD -> development, origin/development) feature: versionamiento

      Esta línea indica que actualmente estas en la rama development (HEAD) y origin/development significa que development también existe en el repositorio remoto

      feature: versionamiento es el nombre del commit

    • La parte amarilla indica el ID del commit (los primeros 7 caracteres del hash).

    • La parte blanca indica el mensaje del commit describiendo los cambios

    • * 9faf9af (origin/main, main) initial commit

      9faf9af es el hash del commit, main es la rama principal, origin/main significa que main tambien existe en el repositorio remoto.

    Resumen

    main es la rama principal (origin/main, main). development es una rama creada desde main y está más adelantada. Se han hecho varios commits en development con cambios en trainee, headbar y scroll. El último commit (f9caa50) incluye versionamiento. development está sincronizado con origin/development.

  • git add

    Añade los archivos modificados al área de preparación (staging area). Los archivos modificados se mueven de la carpeta de trabajo a la zona de preparación, para ser incluidos en el siguiente commit.

    Si usamos git add . se anadirán todos los archivos modificados en la carpeta de trabajo.

    Si usamos git add nombre-del-archivo se anadirán todos los archivos modificados en la carpeta de trabajo y subcarpetas.

    Despues de anadir los archivos, puedes usar git status para revisar el estado de tu proyecto.

  • git commit

    Confirma los cambios realizados en la zona de preparación (staging area), lo ideal es usar un mensaje corto y claro para que el commit sea legible.

    En mi caso personal primero pongo el tipo de cambio que se subirá y luego la descripcion del cambio.

    Por ejemplo git commit -m "feature: nuevo componente versionamiento"

    Si quieres modificar el último commit (y aun no has usado git push) puedes usar git commit --amend -m "feature: nuevo componente versionamiento y funcionalidad de versionamiento"

  • git push

    Envía el contenido de la zona de preparación (staging area) hacia el repositorio remoto.

    Si usamos git push subimos los cambios en la rama que estamos trabajando.

    Si usamos git push origin development subimos los cambios en la rama development en el repositorio remoto.

  • git pull

    Si trabajas en equipo es importante tener las ramas en sincronización, para ello usamos git pull, este comando descarga los cambios realizados en el repositorio remoto y los incorpora en la rama local.

  • git branch

    Lista todas las ramas del repositorio.

  • git checkout

    En git checkout tenemos varios usos:

    • git checkout nombre-rama: Cambia de rama.

    • git checkout -b nombre-rama: Crea una nueva rama.

    • git checkout -b nombre-rama origin/rama-remota: Crea una nueva rama y la sincroniza con la rama remota.

    • git checkout -- nombre-del-archivo Si hiciste cambios en un archivo pero aún no los has agregado (git add), puedes restaurarlo a su última versión confirmada

    • git checkout commit -- archivo: Si necesitas traer un archivo específico desde un commit anterior

    • git checkout commit: Si necesitas ver el estado de tu proyecto en un commit antiguo sin alterar la rama actual

    • git checkout -b nueva-rama commit: Si quieres crear una nueva rama a partir de un commit anterior.

    Alternativas recientes para git checkout:

    git switch nombre-rama Para cambiar de rama.

    git restore nombre-archivo para restaurar un archivo

  • git stash

    Guarda temporalmente los cambios en tu directorio de trabajo sin hacer un commit. Esto es útil cuando necesitas cambiar de rama o actualizar el código sin perder tu progreso actual.

    Guardamos con git stash y recuperamos el último stash con git stash apply borramos el último stash con git stash drop
